Exceptions (Fehlerbehandlung)

try/except/else/finally, raise, assert, Exception-Hierarchie, eigene Exception-Klassen.

7.1 Was sind Exceptions?

Eine Exception ist ein Ereignis, das den normalen Programmfluss unterbricht, wenn etwas Ungewöhnliches passiert — z. B. Division durch 0, Dateizugriff fehlgeschlagen.

Wird eine Exception nicht abgefangen, endet das Programm mit einem Traceback.

7.3 Die Exception-Hierarchie

BaseException
 ├── SystemExit
 ├── KeyboardInterrupt
 ├── GeneratorExit
 └── Exception                       ← fangen wir normalerweise
      ├── ArithmeticError
      │    ├── ZeroDivisionError
      │    ├── OverflowError
      │    └── FloatingPointError
      ├── LookupError
      │    ├── IndexError
      │    └── KeyError
      ├── ValueError
      ├── TypeError
      ├── NameError
      ├── AttributeError
      ├── OSError
      │    └── FileNotFoundError
      └── ImportError
Wichtig: Wenn man except Exception schreibt, werden alle Unterklassen (IndexError, KeyError, …) mit abgefangen. except BaseException fängt sogar Strg+C — meist unerwünscht.

7.4 try / except

try:
    x = int(input("Zahl: "))
except ValueError:
    print("Keine gültige Zahl!")

Mehrere Exception-Typen

try:
    # ...
except ValueError:
    print("ValueError")
except ZeroDivisionError:
    print("Div durch 0")
except (TypeError, NameError):    # mehrere in einem
    print("Typ- oder Namensfehler")
except Exception as e:              # Catch-all (aber nicht BaseException)
    print(f"Allgemeiner Fehler: {e}")
Reihenfolge zählt! Spezifische Exceptions zuerst, generische danach. except Exception würde sonst alles fangen und die spezifischen Blöcke wären unerreichbar.

7.5 else und finally

try:
    x = int("42")
except ValueError:
    print("fehler")
else:
    print("kein Fehler, läuft NUR wenn try erfolgreich")
finally:
    print("Läuft IMMER (Cleanup)")
  • else: läuft wenn kein Fehler im try auftrat
  • finally: läuft immer (selbst bei return, erneutem raise)

7.6 Das Exception-Objekt

try:
    int("abc")
except ValueError as e:
    print(e)          # Fehlermeldung: "invalid literal for int() ..."
    print(type(e))    # <class 'ValueError'>
    print(e.args)     # ('invalid literal ...',)

7.7 raise — Fehler werfen

def teile(a, b):
    if b == 0:
        raise ZeroDivisionError("Division durch 0!")
    return a / b

# Re-raise in einem except-Block:
try:
    ...
except ValueError:
    print("log...")
    raise                # ohne Argument → wirft den aktuellen erneut

# Neue Exception aus bestehender (Chain):
try:
    int("abc")
except ValueError as e:
    raise RuntimeError("Umwandlung fehlgeschlagen") from e

7.8 assert

def wurzel(x):
    assert x >= 0, "x darf nicht negativ sein"
    return x ** 0.5

wurzel(4)      # 2.0
wurzel(-1)     # AssertionError: x darf nicht negativ sein
Nicht für Input-Validierung im produktiven Code: python -O (Optimize-Modus) deaktiviert alle assert. assert ist nur für Invarianten und Debugging.

7.9 Eigene Exception-Klassen

class MeinFehler(Exception):
    pass

class ValidierungsFehler(Exception):
    def __init__(self, feld, grund):
        super().__init__(f"{feld}: {grund}")
        self.feld = feld
        self.grund = grund

try:
    raise ValidierungsFehler("email", "ungültig")
except ValidierungsFehler as e:
    print(e.feld, e.grund)

Neue Exception-Klassen sollten von Exception erben.

7.10 Best Practices

  • Spezifisch fangen: except ValueError besser als except Exception.
  • Nicht schlucken: ein leerer except: pass versteckt Fehler.
  • finally für Cleanup: Dateien schließen, Verbindungen beenden.
  • Fehler propagieren lassen, wenn du sie nicht sinnvoll behandeln kannst.
  • EAFP („Easier to Ask Forgiveness than Permission”) ist idiomatisch: erst versuchen, dann fangen — statt vorher zu prüfen.
