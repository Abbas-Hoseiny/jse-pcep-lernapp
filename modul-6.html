Fehler, Debugging & Troubleshooting

Fehlerklassen, try/catch/finally, throw, Debugging-Strategien.

6.1 Fehlerarten

Drei große Kategorien:

  1. Syntaxfehler — der Code kann gar nicht erst geparst werden (z. B. fehlende Klammer).
  2. Laufzeitfehler (Runtime Errors, auch „Exceptions") — treten während der Ausführung auf.
  3. Logische Fehler — der Code läuft, macht aber das Falsche.

6.2 Fehlerklassen in JavaScript

  • Error — Basisklasse
  • SyntaxError — ungültige Syntax (z. B. JSON.parse("{"))
  • ReferenceError — Variable existiert nicht: console.log(x) ohne Deklaration
  • TypeError — Operation auf falschem Typ: null.foo
  • RangeError — Wert außerhalb des gültigen Bereichs: new Array(-1)
  • URIError — bei decodeURI-Problemen
  • EvalError — historisch, selten
console.log(x);                // ReferenceError: x is not defined
null.foo;                       // TypeError: Cannot read property of null
[].length = -1;                 // RangeError: Invalid array length
JSON.parse("{");                // SyntaxError: Unexpected end of JSON

6.3 try / catch / finally

try {
  // Code, der fehlschlagen KANN
  let x = JSON.parse(input);
} catch (err) {
  // Wird ausgeführt, WENN ein Fehler geworfen wurde
  console.log("Fehler:", err.message);
} finally {
  // Wird IMMER ausgeführt — mit oder ohne Fehler
  console.log("Aufräumen");
}

Reihenfolge:

  1. try-Block läuft.
  2. Wird ein Fehler thrown, springt die Ausführung in catch.
  3. finally läuft immer — auch bei return oder erneutem throw.
Wichtig:
  • catch oder finally ist Pflicht (mindestens einer der beiden).
  • Moderne Syntax: catch { } ohne Parameter möglich (optional binding).

6.4 Fehler werfen mit throw

function teile(a, b) {
  if (b === 0) throw new Error("Division durch 0!");
  return a / b;
}

try {
  teile(10, 0);
} catch (err) {
  console.log(err.name);     // "Error"
  console.log(err.message);  // "Division durch 0!"
}

// Eigene Fehlerklassen:
class ValidationError extends Error {
  constructor(msg) {
    super(msg);
    this.name = "ValidationError";
  }
}
throw new ValidationError("Ungültige Eingabe");

Man kann alles werfen, aber Konvention ist Error-Instanz:

throw "einfacher String";     // funktioniert, aber unüblich
throw { code: 42 };            // auch erlaubt
throw new Error("Besser!");    // empfohlen

6.5 Das Error-Objekt

const err = new Error("Etwas ging schief");
err.name;      // "Error"
err.message;   // "Etwas ging schief"
err.stack;     // Call-Stack (Debug-Info)

// Subklassen:
new TypeError("falscher Typ");
new RangeError("außer Bereich");

6.6 Debugging-Strategien

  • console.log(wert) — der Klassiker
  • console.table(arr) — schöner Output für Arrays/Objekte
  • console.error(), console.warn() — farblich markiert
  • console.dir(obj) — tiefer Einblick in ein Objekt
  • debugger; im Code → der Browser-Debugger stoppt dort
  • Breakpoints im DevTools → Sources-Tab
  • Call-Stack prüfen: wer hat die Funktion aufgerufen?
  • Schrittweise ausführen (Step Over / Step Into)
function f(x) {
  debugger;         // Browser stoppt hier
  return x * 2;
}

6.7 Strict Mode

"use strict";

x = 10;           // ReferenceError (nicht deklariert)
delete Object.prototype;  // TypeError (statt silent fail)

Strict Mode macht JS strenger und hilft beim Debugging. Er ist Standard in Modulen und class-Bodies.
